# Admin Deployment Guide

## Problem

In a team/enterprise setting, individual users should not:
- Create their own OAuth apps or service accounts
- See org-wide API tokens (Google, Jira, Slack, etc.)
- Choose which integrations are available to them
- Need to run the interactive setup wizard

The admin configures everything centrally and deploys scoped bundles to
each machine or user.

## Concepts

### Manifest (`phynai-manifest.yaml`)

The admin's source of truth. Lives on the admin workstation only — never
deployed to target machines. Contains:

- **credentials**: All org-wide API keys, tokens, OAuth refresh tokens
- **roles**: Named permission sets (e.g. `sre`, `developer`, `readonly`)
  - Each role lists which integrations it can access
  - Each role specifies risk ceiling (e.g. `readonly` → `Risk.LOW` only)
  - Each role can override agent settings (model, max iterations)
- **hosts**: Maps machines/users to roles

### Bundle

A directory generated by `phynai admin provision` for a specific role.
Contains only the credentials and config that role needs. Ready to SCP
to a target machine's `~/.phynai/`.

```
bundle-sre/
  config.yaml       # provider, model, agent settings
  .env              # only credentials for tools this role can access
  policy.yaml       # tool whitelist, risk ceiling, confirmation rules
```

### Policy (`policy.yaml`)

Controls what the agent can do on a target machine:

```yaml
# Which tools are available
tools:
  allow:
    - terminal
    - file_tools
    - web_tools
    - github
    - jira
    - pagerduty
  deny: []           # explicit deny overrides allow

# Risk ceiling — agent cannot execute tools above this level
risk_ceiling: high   # low | medium | high | critical

# Which tools require human confirmation even if under ceiling
require_confirmation:
  - terminal.execute   # always confirm shell commands

# Credential source — where .env secrets come from
credential_source: bundle   # bundle | vault | env
```

## Workflow

### 1. Admin initializes the manifest

```bash
phynai admin init
```

Creates `phynai-manifest.yaml` with a scaffold. The admin fills in
credentials and defines roles.

### 2. Admin provisions bundles

```bash
# Generate bundle for a specific role
phynai admin provision --role sre --output ./bundles/sre/

# Generate bundles for all roles
phynai admin provision --all --output ./bundles/
```

Each bundle directory contains only what that role needs. Credentials
for integrations not in the role's `allow` list are excluded.

### 3. Admin deploys to target machines

```bash
# Deploy to a single host
phynai admin deploy --bundle ./bundles/sre/ --host 10.0.1.50 --user ops

# Deploy to hosts from manifest
phynai admin deploy --bundle ./bundles/sre/ --hosts-from-manifest

# Deploy all roles to all assigned hosts
phynai admin deploy --all
```

Under the hood this is `scp` + `ssh` (or can output an Ansible playbook).

### 4. Users run phynai — no setup needed

```bash
phynai chat
```

Config and credentials are pre-populated. The agent loads only the tools
allowed by the policy. Users can still run `phynai setup agent` to change
personal preferences (log level, max iterations) but cannot modify
credentials or tool access.

## Manifest Format

```yaml
# phynai-manifest.yaml — admin eyes only
version: 1

# ── Organization defaults ───────────────────────────────
defaults:
  provider: anthropic
  model: claude-sonnet-4-6
  max_iterations: 50
  log_level: warning

# ── Credentials ─────────────────────────────────────────
# All org-wide secrets. These never leave the admin machine
# as-is — they're sliced per-role during provisioning.
credentials:
  provider:
    ANTHROPIC_API_KEY: sk-ant-...
  google:
    GOOGLE_CLIENT_ID: "123456.apps.googleusercontent.com"
    GOOGLE_CLIENT_SECRET: "GOCSPX-..."
    GOOGLE_REFRESH_TOKEN: "1//0e..."
  github:
    GITHUB_TOKEN: "ghp_..."
  jira:
    JIRA_URL: "https://yourcompany.atlassian.net"
    JIRA_EMAIL: "svc-phynai@company.com"
    JIRA_API_TOKEN: "ATATT3..."
  slack:
    SLACK_BOT_TOKEN: "xoxb-..."
    SLACK_APP_TOKEN: "xapp-..."
  ms365:
    MICROSOFT_TENANT_ID: "..."
    MICROSOFT_CLIENT_ID: "..."
    MICROSOFT_CLIENT_SECRET: "..."
  okta:
    OKTA_ORG_URL: "https://company.okta.com"
    OKTA_API_TOKEN: "00..."
  pagerduty:
    PAGERDUTY_API_KEY: "..."

# ── Roles ───────────────────────────────────────────────
roles:
  sre:
    description: "Site reliability engineers — full ops access"
    integrations:
      - provider
      - github
      - jira
      - slack
      - pagerduty
      - okta
    tools:
      allow: [terminal, file_tools, web_tools, github, jira, slack, pagerduty, okta]
    risk_ceiling: high
    require_confirmation: [terminal.execute]
    overrides:
      max_iterations: 100

  developer:
    description: "Software engineers — code + ticketing"
    integrations:
      - provider
      - github
      - jira
    tools:
      allow: [terminal, file_tools, web_tools, github, jira]
    risk_ceiling: medium
    require_confirmation: [terminal.execute]

  readonly:
    description: "Read-only access for auditors and managers"
    integrations:
      - provider
      - github
      - jira
    tools:
      allow: [file_tools, web_tools, github, jira]
      deny: [terminal]
    risk_ceiling: low
    overrides:
      model: claude-haiku-4-5
      max_iterations: 20

  analyst:
    description: "Data analysts — SQL + Google + files"
    integrations:
      - provider
      - google
    tools:
      allow: [file_tools, web_tools, google, sql]
    risk_ceiling: medium

# ── Host assignments ────────────────────────────────────
hosts:
  # hostname-or-IP: role
  10.0.1.50: sre
  10.0.1.51: sre
  10.0.2.10: developer
  10.0.2.11: developer
  10.0.3.5: readonly
  # Or by user@host
  alice@workstation-01: developer
  bob@workstation-02: analyst
```

## Security

- **Manifest stays on admin machine.** Never deployed, never committed.
  Add to `.gitignore`.
- **Bundles contain only scoped secrets.** A `developer` bundle has no
  Google, Slack, or Okta credentials.
- **File permissions.** `.env` files are written with `chmod 0600`.
  `policy.yaml` is `0644` (not secret).
- **Rotation.** Re-run `phynai admin provision` + `deploy` after
  rotating any credential. The manifest is the single source of truth.
- **Vault integration (future).** Instead of embedding secrets in the
  manifest, point to Vault paths. The provisioner fetches at bundle time.
  Target machines can also pull directly from Vault at runtime.

## Future: `credential_source: vault`

```yaml
credentials:
  github:
    _vault: "secret/data/phynai/github"
  jira:
    _vault: "secret/data/phynai/jira"
```

When `_vault` is present, the provisioner calls Vault's KV API to
resolve the actual values at bundle-generation time. For runtime Vault
access (no secrets on disk at all), set `credential_source: vault` in
the policy and the agent fetches tokens on startup.

## CLI Reference

```
phynai admin init                          Create manifest scaffold
phynai admin validate                      Check manifest for errors
phynai admin provision --role <name>       Generate bundle for role
phynai admin provision --all               Generate all role bundles
phynai admin deploy --bundle <dir> --host  Push bundle to one host
phynai admin deploy --all                  Push all bundles per manifest
phynai admin rotate --credential <name>    Update credential + redeploy
phynai admin audit                         Show who has what access
```
